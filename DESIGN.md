# SAGA 설계 기록

> **TL;DR** — AI 캐릭터와의 장기 대화에서 프롬프트 캐시를 깨지 않고 맥락에 맞는 기억을 회상하는 리버스 프록시.
> 용도별 **3계층 저장소**(SQLite · ChromaDB · .md prefix)와 **동기/비동기 분리**로
> 캐시 적중률 **84.2%**, 순비용 **67.1% 절감**을 확보하면서
> LOCOMO **+54%**, LongMemEval **+42.3%p**를 달성했다.

| 기간 | 유형 | 역할 |
| --- | --- | --- |
| 2026.02 – 개선 중 | 개인 프로젝트 | 1인 개발 (설계·구현·평가) |

---

## 1. 문제 정의

### 1.1 왜 "장기 기억"이 문제인가

AI 캐릭터 챗은 엔터테인먼트 산업에서 빠르게 성장하는 영역이지만, 대화가 길어질수록 세 가지 문제가 반복해서 나타난다.

- **성격 희석** — 30턴을 넘어가면서 캐릭터의 말투·태도·가치관이 평균적인 LLM 어시스턴트에 수렴
- **관계 리셋** — "우리가 어제 싸운 일" 같은 관계 상태가 휘발, 유저가 매번 설명해야 함
- **서사 유실** — 복선, 장소 이동, 등장 NPC 같은 구조적 맥락이 attention 범위를 벗어나면 사실상 소실

컨텍스트 윈도우에 전부 넣어도 해결되지 않는다. 200K 컨텍스트 모델조차 중간 부분의 정보를 소실하는 *lost-in-the-middle* 현상이 있기 때문이다.
단순히 "더 많이 집어넣기"는 답이 아니다.

### 1.2 기존 솔루션의 한계

| 솔루션 | 핵심 한계 |
| --- | --- |
| **HypaMemory V3** (RisuAI) | 매 턴 관련 기억을 system 메시지 중간에 주입 → prefix 해시 변경 → **프롬프트 캐시 매 턴 깨짐**. 실측 캐시 히트 12% 수준 |
| **Letta** (MemGPT 계열) | 자율 에이전트형 메모리로 턴당 **3~5회 LLM 호출** (조회·업데이트·요약·결정). 레이턴시·비용 폭증 |
| **mem0** | 범용 Fact 저장에 최적화. NPC 관계·이벤트 체인 같은 **서사 구조를 표현하기 어려움** |

기존 솔루션들은 "기억을 담는 방법"을 고민했지만 **"기억을 어떻게 집어넣을 때 캐시가 살아남는가"**를 함께 고민하지 않았다.
내가 주목한 것은 이 지점이었다.

> **KEY INSIGHT**
> "캐릭터가 살아있다는 감각"의 본질은 **기억의 양**이 아니라 **맥락에 맞는 회상**이다.
> 그리고 회상을 잘하는 것과 별개로, **매 턴 컨텍스트를 재조립하는 비용이 서비스 생존을 결정한다.**
> 이 두 축을 동시에 풀어야 진짜 장기 기억이 가능하다.

## 2. 아키텍처 설계

### 2.1 설계 원칙

- **캐시 불변성** — prefix 영역의 바이트 해시는 세션 동안 변하지 않는다. 동적 컨텍스트는 캐시 경계 바깥에만 주입
- **단일 동기 호출** — 유저 응답 경로에는 메인 LLM 1회만. 기억 갱신은 전부 응답 후 백그라운드
- **용도별 저장소 분리** — 구조적 조회 · 의미 검색 · prefix 직렬화는 요구 특성이 달라 단일 저장소로 합치지 않는다
- **요약 1회 → 다용도 재활용** — 매 턴 생성되는 서사 요약을 압축·색인·검색 시드로 동시에 사용

### 2.2 전체 파이프라인

```
═══ CACHE PREFIX (불변) ════════════════════════════════
  [System Prompt] → [.md prefix: 세계관/NPC/상태] → [과거 summary chunks]
═══════════════════════════════════════════════════════

  ▼ 동적 영역 (매 턴 변경)
  [검색 결과 prepend] + [최근 user 메시지]

  ▼ 메인 LLM 1회 호출 (유저 응답 경로)
  [응답 반환 → 유저]

  ▼ 백그라운드 (응답 후)
  ├─ Sub-B 서사 요약 (경량 모델 1회)
  │  └─ 4필드 구조: scene / relationship / fact / foreshadow
  ├─ SQLite 업데이트 (관계·이벤트·Lorebook)
  ├─ ChromaDB 색인 (에피소드 임베딩)
  ├─ MessageCompressor (임계치 초과 시 chunk 압축)
  └─ Curator (모순 검증·lore 생성)
```

유저 체감 레이턴시 = 메인 LLM 1회. 백그라운드 작업은 체감 경로 밖.

### 2.3 저장소 3계층

| 저장소 | 용도 | 설명 |
| --- | --- | --- |
| **SQLite** | 구조적 사실 | 관계 그래프, 이벤트 체인, Lorebook(단일 소스), 세계관 상태. "A와 B의 현재 관계" 같은 정확한 조건 조회에 사용 |
| **ChromaDB** | 에피소드 회상 | 대화 턴을 에피소드 단위로 임베딩 저장. 의미 유사도 기반 Top-K 검색. "숲에서 있었던 일" 같은 fuzzy 질의에 강함 |
| **.md prefix** | 프롬프트 캐싱 | 세계관·현재 상태를 Markdown으로 직렬화. Anthropic 3-Breakpoint 캐싱 적용. prefix 해시 안정 → 캐시 유지 |

단일 벡터 DB로 모든 걸 처리하면 구조적 사실 조회가 느리고, 단일 구조화 DB로만 처리하면 에피소드 회상이 불가능하다.
각 저장소는 서로 다른 쿼리 특성에 답하며, 필요에 따라 교차 결합된다 (예: ChromaDB에서 찾은 에피소드의 등장 NPC를 SQLite로 역조회).

## 3. 구현하면서 부딪힌 문제들

### 3.1 동적 컨텍스트 주입 위치

매 턴 검색된 기억(관련 에피소드·관계 변화)을 프롬프트 어디에 넣을지는 처음엔 간단한 문제로 보였다.
지시문 성격이니 system 메시지 끝에 붙이면 되겠다고 생각했다. 실제로 첫 프로토타입은 그렇게 구현했다.

문제는 캐시 로그를 찍어보면서 드러났다. cache_read 비율이 10% 초반에 머물렀다.
Anthropic Prompt Caching은 prefix 바이트가 1개만 바뀌어도 무효화되는데, system 메시지에 매 턴 다른 검색 결과를 넣으니 **캐시 포인트 앞 전체가 재계산**되고 있었다.
검색 결과는 user 메시지 앞에 prepend하는 걸로 위치를 옮겼다. user 메시지는 어차피 매 턴 바뀌는 영역이라 캐시 경계 바깥이다.

우려했던 품질 저하는 실험에서 유의미하지 않았다. 유저 발화 직전에 "참고 정보"로 붙는 편이 오히려 자연스럽게 반영되는 경향도 있었다.
이 한 번의 위치 변경으로 cache_read 비율이 84%까지 올라갔다.

### 3.2 오래된 대화를 어떻게 압축할 것인가

세션이 길어지면 컨텍스트가 폭증한다. 가장 단순한 해법은 매 요청마다 과거 히스토리를 요약 모델로 돌려 압축본을 만드는 것이지만, 이러면 **요약 결과가 매번 미세하게 달라지면서 prefix 해시가 불안정해진다**. 캐시를 깨려고 압축하는 꼴이 된다.

대신 `MessageCompressor`를 만들어, 과거 턴을 [user+assistant] 쌍으로 묶어 한 번 요약한 뒤 **immutable chunk로 동결**하는 방식으로 구현했다.
한번 만든 chunk는 변경되지 않으므로 prefix는 안정적이다. 요약 내용은 이미 Sub-B가 매 턴 생성하는 4필드 서사 요약을 그대로 재활용해 LLM 호출을 추가로 쓰지 않았다.

구현 디테일 두 가지가 꽤 까다로웠다.

- 캐시 브레이크포인트(BP2)를 **첫 번째 chunk의 assistant 메시지에 고정**. 새 chunk가 추가돼도 BP2 위치가 흔들리지 않도록.
- 압축 임계치 판단은 `rebuild-first` — 기존 chunk가 있으면 먼저 재조립한 뒤 **실제 토큰 수**로 판단. 원본 히스토리 기준으로 판단하면 이미 압축된 대화를 또 압축하는 오버 컴프레션이 발생했다.

### 3.3 서사 요약을 한 번만 생성하기

파이프라인을 처음 짤 때는 압축용·검색 시드용·색인용·Curator 검증용으로 요약이 각각 필요했다.
별 생각 없이 각자 독립 생성했더니 턴당 경량 모델 호출이 4번씩 쌓였다.
누적되면 무시할 수 없는 비용이고, 더 문제는 **같은 턴의 내용을 각자 다르게 요약**하면서 관계 업데이트와 에피소드 색인이 미묘하게 안 맞는 케이스가 나오기 시작했다.

Sub-B(Post-Turn Agent)에서 4필드 구조로 한 번만 생성하고, 네 곳에서 이걸 공유하는 구조로 바꿨다.
단일 진실원이 생기면서 일관성 문제가 사라졌고 LLM 호출도 1/4로 줄었다.

```python
# Sub-B 출력 구조 — 한 번 생성, 네 곳에서 공유
class TurnSummary:
    scene:        "장면 요약 — 위치·시간·주요 사건"
    relationship: "NPC 관계 변화 — 델타만"
    fact:         "새로 확정된 세계관 사실"
    foreshadow:   "언급된 복선·미해결 떡밥"

# 소비처: MessageCompressor chunk / ChromaDB 색인 /
#         SQLite 관계 업데이트 / Curator 모순 검증
```

## 4. 성과 및 평가

### 4.1 핵심 지표

| 지표 | 결과 | 비고 |
| --- | --- | --- |
| LOCOMO | **+54%** | judge 2.02 → 3.12, multi-hop 4.71/5 |
| LongMemEval | **+42.3%p** | 21.2% → 63.5% |
| 프롬프트 캐시 적중률 | **84.2%** | steady-state 86.8% |
| 순비용 절감 | **67.1%** | cache_read 단가 + 프롬프트 단축 합산 |

### 4.2 벤치마크 방법

- **LOCOMO (ACL 2024)** — 장기 대화 QA 벤치마크. 단일 세션 내 50~100턴에서 과거 사실 회상 능력을 측정. Judge 점수(1~5)로 평가
- **LongMemEval (ICLR 2025)** — 시간 간격을 둔 여러 세션에서 사실·관계·시간 추론 능력 측정
- **multi-hop 세부 지표** — "A가 B에게 준 물건을 C가 알고 있는가" 같은 2-hop 이상 질문에서 4.71/5 달성

> **벤치마크의 한계**
> 솔직히 말하면, 캐릭터 챗의 품질을 LOCOMO나 LongMemEval로 온전히 평가하기는 어렵다.
> 이 벤치마크들은 "대화에 포함된 사실을 얼마나 정확히 회상하는가"를 정답이 정해진 QA로 측정하지만,
> 실제 롤플레이는 유저의 한 마디로 장소가 바뀌고 새 NPC가 튀어나오며 기존 설정이 즉흥적으로 갱신되는 **non-stationary 환경**이다.
> 게다가 "캐릭터가 살아있다는 감각"은 회상 정확도 외에도 말투 일관성·관계 밀도·상황 판단 같은 정성 요소가 크게 작용한다.
>
> 그래서 SAGA는 정량 벤치마크를 **"큐레이션 정확도의 하한 체크"**로만 사용하고,
> 실제 품질 판단은 개발자 본인의 장기 세션 도그푸딩(동일 캐릭터로 수백 턴 누적 테스트)에 의존한다.
> 외부 유저 피드백 루프가 없다는 점은 이 프로젝트의 명확한 한계이며, 향후 소규모 클로즈드 베타로 해소할 계획이다.
> 벤치마크 점수 개선이 곧 UX 개선과 동치가 아니라는 점은 이 프로젝트를 통해 체감한 중요한 교훈이기도 하다.

### 4.3 캐시·비용 분석

Anthropic API의 cache_read 토큰과 cache_write 토큰을 분리 집계하여 실제 청구액 기준으로 비교했다.
SAGA OFF(기본 RisuAI) 대비 ON 상태에서:

- **cache_read 비율 84.2%** — 입력 토큰의 84%가 캐시 히트로 처리되어 1/10 단가 적용
- **steady-state 86.8%** — 세션 초반 5턴을 제외한 안정 구간에서의 수치. 실서비스 조건에 가까움
- **순비용 67.1% 절감** — 캐시 할인 + 프롬프트 단축 효과 합산

### 4.4 정성 평가 (개발자 도그푸딩)

외부 유저 풀이 없어 정성 평가는 개발자 본인이 동일 캐릭터·시나리오로 수백 턴 누적 세션을 반복해 재현성을 확인하는 방식으로 진행했다.
max context 55K~65K, HypaMemory OFF + SAGA ON 설정 기준으로 관찰한 변화는 다음과 같다.

- 30만 토큰 누적 세션에서도 **캐릭터 말투·태도 붕괴 없이 유지** — HypaMemory ON 대비 육안으로 구분될 정도의 차이
- "그때 ○○이 했던 말" 류 과거 참조 질문에 **구체적 에피소드 회상** (기존에는 "기억나지 않는다" 또는 환각 응답이 지배적)
- 응답 레이턴시 체감 변화 없음 — 기억 갱신이 백그라운드로 격리되어 유저 경로에 노출되지 않음

> **한계**
> N=1 평가이며, 개발자 본인이 시나리오를 구성·실행·판정하므로 편향이 있을 수밖에 없다.
> 이 한계를 알고 있고, 외부 유저 대상 블라인드 A/B 테스트가 다음 단계의 과제다.

## 5. 현재 진행 및 로드맵

### 5.1 진행 중인 리팩터링

- **chat_handler.py 책임 분리** (완료) — 너무 많은 기능이 한 파일에 뭉쳐 있던 512줄짜리 모듈을 세션 관리·캐시 마킹·파이프라인 오케스트레이션·후처리 4파일로 분해
- **WindowRecovery 제거** (완료) — MessageCompressor가 완전 대체하면서 파이프라인 단계 1개 축소
- **ChromaDB 용도 재정렬** (완료) — Lorebook을 SQLite 단일 소스로 이관, ChromaDB는 에피소드 전용으로 역할 명확화
- **Curator 책임 분리** (진행 중) — 모순 검증과 lore 생성을 별도 모듈로 분해

### 5.2 미해결 이슈 및 개선 계획

- **NPC 누적 문제** — 100턴 이상 세션에서 엑스트라 NPC가 누적되며 prefix가 비대화. 중요도 기반 필터링 전략 설계 중
- **숫자 모순 탐지** — Sub-B가 가격·수량 같은 구체 숫자를 추출하지 않아 Curator가 숫자 불일치를 잡지 못함. 요약 스키마 확장 검토
- **세션 자동 분리** — 현재는 수동(x-saga-session-id 헤더). Lorebook 변경 시 자동 분기 필요
- **RisuAI 플러그인 배포** — 리버스 프록시 외에도 네이티브 플러그인 프로바이더 형태로 배포 검토

### 5.3 경쟁 제품 분석과 흡수 검토

같은 문제 영역을 다루는 [**RisuAI Agent plugin v5.3.1**](https://github.com/EugenesDad/RisuAI-Agent-plugin)의 구조를 분석하면서,
해당 제품이 SAGA보다 앞서 있는 한 가지 설계를 확인했다. 프롬프트에 주입하는 정보를
**네임스페이스화된 구조적 객체로 분해하고 LLM에게 우선순위로 해석하도록 지시**하는 방식이다.

RisuAI Agent는 시스템 프롬프트에 `ra_*` 접두사를 붙인 20여 개 객체를 주입하고 5단계 우선순위를 명시한다.
일부 예시만 발췌하면 다음과 같다.

| 단계 | 객체 | 역할 |
| --- | --- | --- |
| P1 (강제) | `ra_logic_state`, `ra_response_guard`, `ra_pattern_guard` | 금지 표현·강제 지시·최우선 오버라이드 |
| P2 (장면) | `ra_scene_state`, `ra_inventory`, `ra_turn_trace` | 현재 위치·소지품·직전 비트 |
| P3 (플롯) | `ra_quest_log`, `ra_knowledge_matrix`, `ra_relation_web` | 진행 중 사건·정보 경계·관계 |
| P4 (연속성) | `ra_persistent_memory`, `ra_reentry_guard`, `ra_arc_memory` | 정체성·재등장 복구·서사 아크 |
| P5 (참조) | `ra_world_encyclopedia`, `ra_world_log` | 필요 시에만 참조 |

**현재 SAGA의 한계** — `.md prefix`는 "세계관 / 캐릭터 / 현재 상태" 수준의 단일 Markdown 문서로 직렬화되어 있어,
LLM이 충돌을 만났을 때 무엇을 우선시할지 판단 근거가 부족하다. 실제 도그푸딩 중
"금지된 상황인데 캐릭터가 어쩔 수 없이 따라가는" 경계 실패 케이스가 종종 발생했다.

**흡수 검토 방향** — SAGA의 캐시 불변성 원칙을 해치지 않는 선에서 다음 3가지를 적용 예정.

- **네임스페이스 라벨링** — `.md prefix`에 섹션별 라벨(예: `@@scene_state`, `@@persistent_memory`)을 도입. 캐시 prefix 자체는 그대로 유지하되 LLM이 구조를 인지하도록
- **우선순위 가이드 주입** — system 프롬프트 상단에 "섹션 간 충돌 시 우선순위" 규칙을 1회 고정 주입 (prefix 일부로 편입되므로 캐시 유지)
- **동적 영역에 `turn_advice` 도입** — user prepend 구간에 이번 턴 연기 가이드(`primary_facet`, `response_guard`)를 주입. 캐시 경계 바깥이므로 안전

> **출처 명시**
> 위 네임스페이스 설계는 RisuAI Agent plugin (저자: penguineugene@protonmail.com, GPL-3.0)의 공개 플러그인 번들을 구조 분석해 파악한 패턴이다.
> SAGA는 이 설계 아이디어만 참고하며, 구현 코드는 독립적으로 작성한다.
> 해당 플러그인은 브라우저 환경에서 턴당 4~6회 LLM 호출을 사용하는 반면, SAGA는 동일한 구조 가이드를 **캐시 불변 prefix + 비동기 단일 호출** 조합으로 구현해 비용 구조의 차이를 유지할 계획이다.

## 6. 프로젝트를 통해 배운 것

이 프로젝트를 하면서 얻은 가장 큰 학습은 **"AI 시스템의 성능은 모델 품질만이 아니라 그 모델을 둘러싼 데이터 흐름 설계에 의해 결정된다"**는 것이었다.
같은 LLM을 쓰더라도 캐시 경계를 어디에 긋느냐, 어떤 작업을 동기·비동기로 분리하느냐에 따라 실제 서비스의 레이턴시와 비용은 3~5배 차이가 난다.

또한 **"벤치마크 점수 = 사용자 경험"이 아니라는 것**도 체감했다.
LOCOMO에서 +54%를 올렸지만 실사용자가 체감하는 개선은 "캐릭터가 더 자연스럽게 기억하는 것" 쪽이었고,
이 둘 사이의 갭을 메우려면 정량 지표 바깥의 정성 피드백 수집 루프가 필요했다.

마지막으로, **한 모듈이 너무 많은 일을 하지 않도록 책임을 잘게 쪼개는 작업이 AI 파이프라인에서도 그대로 효과가 있다**는 것을 배웠다.
Sub-A · Sub-B · Curator · MessageCompressor 각각이 한 가지 일만 하도록 정리한 뒤로 디버깅 시간이 절반 이하로 줄었고, 새 실험을 붙이기가 훨씬 쉬워졌다.
