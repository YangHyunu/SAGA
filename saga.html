PROJECT PORTFOLIO

SAGA

Stateful Context Engine for Narrative Roleplay
기간 2026.02 – 개선 중 유형 개인 프로젝트 역할 1인 개발 (설계·구현·평가)
TL;DR
AI 캐릭터와의 장기 대화에서 프롬프트 캐시를 깨지 않고 맥락에 맞는 기억을 꺼내 쓰는 리버스 프록시. 용도별 3계층 저장소(SQLite · ChromaDB · 프롬프트 앞부분) 와 동기/비동기 분리로 캐시 적중률 84.2%, 순비용 67.1% 절감을 확보하면서 LOCOMO +54%, LongMemEval +42.3%p의 벤치마크 향상을 달성했다. RisuAI·SillyTavern 등 기존 롤플레이 프론트엔드 수정 없이 리버스 프록시로 삽입 가능.
1문제 정의
1.1 장기 기억의 문제

AI 캐릭터 챗은 엔터테인먼트 산업에서 빠르게 성장하는 영역이지만, 대화가 길어질수록 세 가지 문제가 반복해서 나타난다.

  • 성격 희석 — 30턴을 넘어가면서 캐릭터의 말투·태도·가치관이 평균적인 LLM 어시스턴트에 수렴
  • 관계 리셋 — "우리가 어제 싸운 일" 같은 관계 상태가 휘발, 유저가 매번 설명해야 함
  • 서사 유실 — 복선, 장소 이동, 등장 NPC 같은 구조적 맥락이 attention 범위를 벗어나면 사실상 소실

컨텍스트 윈도우에 전부 넣어도 해결되지 않는다. 200K 컨텍스트 모델조차 중간 부분의 정보를 소실하는 lost-in-the-middle 현상이 있기 때문이다. 즉, 단순히 "더 많이 집어넣기"는 답이 아니다.

1.2 기존 솔루션의 한계

시장에는 HypaMemory V3, Letta, mem0 등 여러 메모리 솔루션이 있지만 공통적으로 두 가지 구조적 한계를 가진다.

솔루션 핵심 한계
HypaMemory V3 (RisuAI) 매 턴 관련 기억을 system 메시지 중간에 주입 → 프롬프트 앞부분이 매 턴 달라져 캐시가 깨짐. 실측 캐시 히트 12% 수준
Letta (MemGPT 계열) 자율 에이전트형 메모리로 턴당 3~5회 LLM 호출 (조회·업데이트·요약·결정). 레이턴시·비용 폭증
mem0 범용 Fact 저장에 최적화. NPC 관계·이벤트 체인 같은 서사 구조를 표현하기 어려움

즉, 기존 솔루션들은 기억 저장 방식만 고민했고, 캐시 보존은 별도 문제로 봤다. SAGA는 두 문제를 하나의 설계로 풀었다.

유저가 캐릭터가 살아있는 것처럼 느끼게 하는 것은 기억의 양이 아니라 지금 맥락에 맞는 적절한 기억의 큐레이션에서 나온다. 그리고 큐레이션을 잘하는 것과 별개로, 매 턴 컨텍스트를 재조립하는 비용이 서비스 생존을 결정한다. 이 두 축을 동시에 풀어야 진짜 장기 기억이 가능하다.
2아키텍처 설계
▼ 먼저 이해할 전제 · 프롬프트 캐시 Anthropic API는 프롬프트의 앞부분이 이전 요청과 완전히 동일하면 그 부분을 재계산하지 않고 1/10 가격으로 재사용한다. 조건은 단 하나 — 앞부분의 바이트가 단 1개라도 바뀌면 캐시는 무효가 된다. 이 제약이 SAGA 설계 전반을 관통한다. "기억을 어디에 넣을 것인가" 는 결국 "어디까지 안 바꿀 수 있는가" 의 문제가 된다.
2.1 설계 원칙
  • 앞부분은 건드리지 않는다 — 프롬프트의 고정 영역은 세션 내내 같은 바이트를 유지. 매 턴 바뀌는 정보는 모두 뒤쪽(사용자 발화 앞)에만 배치
  • 유저 응답 경로엔 메인 LLM 1회만 — 기억 갱신·색인·요약은 전부 응답 이후 백그라운드로 밀어냄
  • 저장소는 용도별로 쪼갠다 — 정확한 조회, 의미 검색, 프롬프트 직렬화는 각각 요구가 달라 한 저장소에 합치지 않음
  • 요약은 한 번만 만든다 — 매 턴 생성되는 서사 요약을 압축·색인·검색·검증 네 곳이 모두 공유
2.2 전체 파이프라인
═══ 프롬프트 앞부분 · 세션 동안 바뀌지 않음 ═══════════════ [System Prompt] → [세계관·NPC·상태 요약] → [과거 대화 요약 블록들] ═════════════════════════════════════════════════════════ ▼ 매 턴 바뀌는 뒷부분 (캐시 바깥) [관련 기억 검색 결과] + [유저 새 발화] ▼ 메인 LLM 1회 호출 (유저 응답 경로) [응답 반환 → 유저] ▼ 백그라운드 (응답 후, 유저는 이미 응답 받음) ├─ Sub-B(서사 요약 에이전트) — 경량 모델 1회 │ └─ 4필드 요약: 장면 / 관계 변화 / 사실 / 복선 ├─ SQLite 업데이트 (관계·이벤트·Lorebook) ├─ ChromaDB 색인 (에피소드 임베딩) ├─ 대화 압축 (임계치 초과 시 요약 블록 추가) └─ Curator(모순 검증 에이전트) ※ 유저가 체감하는 대기 시간 = 메인 LLM 1회. 백그라운드 작업은 체감 경로 밖.

3구현하면서 부딪힌 문제들
3.1 검색 결과를 프롬프트 어디에 넣을까

매 턴 검색된 기억(관련 에피소드·관계 변화)을 프롬프트 어디에 넣을지는 처음엔 간단한 문제로 보였다. 지시문 성격이니 system 메시지 끝에 붙이면 되겠다고 생각했다. 실제로 첫 프로토타입은 그렇게 구현했다.

문제는 캐시 로그를 찍어보면서 드러났다. 캐시 재사용 비율이 10% 초반에 머물렀다. 검색 결과는 매 턴 다른 내용이 들어오는데, 그걸 캐시가 걸린 앞부분 영역인 system 메시지에 넣으니 매 턴 앞부분이 새로워져서 캐시 전체가 무효화되고 있었다. 검색 결과는 유저 메시지 바로 앞으로 위치를 옮겼다. 유저 메시지는 어차피 매 턴 바뀌는 영역이라 캐시 바깥이다.

우려했던 품질 저하는 실험에서 유의미하지 않았다. 유저 발화 직전에 "참고 정보"로 붙는 편이 오히려 자연스럽게 반영되는 경향도 있었다. 이 한 번의 위치 변경으로 캐시 재사용 비율이 84%까지 올라갔다.

3.2 오래된 대화를 어떻게 밀어낼까

세션이 길어지면 컨텍스트가 폭증한다. 가장 쉬운 해법은 매 요청마다 과거 대화를 요약 모델로 돌려 압축본을 만드는 것이지만, 이러면 요약 결과가 매번 미세하게 달라진다. 같은 대화를 두 번 요약해도 토씨 하나는 바뀌기 마련이고, 그 바뀐 바이트가 앞부분에 들어가면 캐시를 깨려고 압축하는 꼴이 된다.

그래서 "한 번 만든 요약은 다시 안 건드린다"는 원칙으로 접근했다. 오래된 턴을 user·assistant 한 쌍으로 묶어 딱 한 번 요약하고, 그 묶음을 봉인해서 덩어리(블록)로 고정한다. 앞으로 어떤 경우에도 재생성하지 않는다. 요약 내용 자체도 새로 만들 필요가 없었다 — Sub-B가 매 턴 이미 만들어 둔 4필드 요약을 그대로 가져다 쓰면 되기 때문에 LLM 호출이 추가로 들어가지 않는다.

대화가 길어지면 이런 봉인 블록이 계속 누적된다. 아래 그림이 시간 흐름에 따른 프롬프트 모양이다.

[턴 10] [system] [.md] [블록1] 📌캐시 경계 [최근 대화] [턴 30] [system] [.md] [블록1] 📌캐시 경계 [블록2] [블록3] [최근 대화] [턴 100] [system] [.md] [블록1] 📌캐시 경계 [블록2] ... [블록15] [최근 대화] └─ 이 영역 바이트는 턴이 쌓여도 완전히 동일 → 매 턴 캐시 재사용

블록은 뒤로만 추가되고, 한 번 만들어진 블록은 다시 변경되지 않는다. 그래서 캐시 경계를 가장 앞쪽의 블록1에 못 박아두면, 뒤에 블록이 몇 개가 붙든 앞부분의 바이트는 그대로라서 캐시가 유지된다. 이게 이 설계의 핵심이다.

▼ 구현 디테일 (관심 있는 분만)
  • 캐시 경계 위치 고정 — Anthropic 캐시 포인트(BP2)를 "가장 최근 블록"이 아닌 "맨 앞 블록(블록1)의 assistant 메시지"에 고정. 새 블록이 뒤에 붙어도 경계의 바이트 오프셋이 움직이지 않게 하기 위함.
  • 압축 임계치는 조립 후 측정 — "대화가 N 토큰 넘으면 압축" 판단을 원본 히스토리 기준으로 하면, 이미 블록으로 압축된 대화를 또 압축하는 오버 컴프레션이 발생한다. 해결은 현재 프롬프트를 실제로 조립한 뒤 그 결과의 토큰 수로 판단하는 것. 조립된 프롬프트는 이미 작아져 있으므로 임계치에 한참 미달 → 압축 트리거 안 걸림.
3.3 서사 요약을 한 번만 생성하기

파이프라인을 처음 짤 때는 압축용·검색 시드용·색인용·모순 검증용으로 요약이 각각 필요했다. 별 생각 없이 각자 독립 생성했더니 턴당 경량 모델 호출이 4번씩 쌓였다. 누적되면 무시할 수 없는 비용이고, 더 문제는 같은 턴의 내용을 각자 다르게 요약하면서 관계 업데이트와 에피소드 색인이 미묘하게 안 맞는 케이스가 나오기 시작했다.

Sub-B(Post-Turn Agent)에서 4필드 구조로 한 번만 생성하고, 네 곳이 이 요약을 공유하는 구조로 바꿨다. 기준 요약이 하나로 통일되면서 일관성 문제가 사라졌고 LLM 호출도 1/4로 줄었다.

4.2 벤치마크 방법
  • LOCOMO (ACL 2024) — 장기 대화 QA 벤치마크. 단일 세션 내 50~100턴에서 과거 사실 회상 능력을 측정. Judge 점수(1~5)로 평가, SAGA는 judge 2.02 → 3.12로 향상 (+54%)
  • LongMemEval (ICLR 2025) — 시간 간격을 둔 여러 세션에서 사실·관계·시간 추론 능력 측정. SAGA는 21.2% → 63.5% (+42.3%p)
  • multi-hop 세부 지표 — "A가 B에게 준 물건을 C가 알고 있는가" 같은 2-hop 이상 질문에서 4.71/5 달성
벤치마크의 한계 캐릭터 챗의 품질을 LOCOMO와 LongMemEval만으로 온전히 평가하기는 어렵다. 두 벤치마크는 "대화에 포함된 사실을 얼마나 정확히 회상하는가"를 정답이 정해진 QA로 측정하지만, 실제 롤플레이는 유저의 한 마디로 장소가 바뀌고 새 NPC가 튀어나오며 설정이 즉흥적으로 갱신되는 non-stationary 환경이다. 말투 일관성·관계 밀도·상황 판단 같은 정성 요소도 크게 작용한다. 이 한계를 인정하고, SAGA는 정량 벤치마크를 "큐레이션 정확도의 하한 체크"로 사용하며, 품질 판단은 동일 캐릭터로 수백 턴을 누적한 장기 재현성 테스트로 보완한다.
4.3 캐시 · 비용 분석

Anthropic API의 캐시 재사용 토큰과 캐시 쓰기 토큰을 분리 집계하여 실제 청구액 기준으로 비교했다. SAGA OFF(기본 RisuAI) 대비 ON 상태에서:

  • 캐시 재사용 비율 84.2% — 입력 토큰의 84%가 캐시 히트로 처리되어 1/10 단가 적용
  • steady-state 86.8% — 세션 초반 5턴을 제외한 안정 구간에서의 수치. 실서비스 조건에 가까움
  • 순비용 67.1% 절감 — 캐시 할인 + 프롬프트 단축 효과 합산
