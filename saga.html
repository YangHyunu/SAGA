PROBLEM

장기 대화에서 LLM은 맥락을 놓친다

RP 챗봇을 장기 세션으로 굴리다 보면 구조적인 문제에 부딪힌다. 50턴 전에 죽은 NPC가 다시 나타나고, 버린 아이템이 인벤토리에 남아 있고, 동쪽으로 이동했는데 서쪽 마을에 있다. 200K 토큰을 보내도 LLM이 50턴 전 세부사항을 정확히 기억하리라는 보장은 없고, 비용은 선형으로 늘어난다.

// 실제 장기 RP 유저의 현실 max context 60K, HypaMemory ON, 메토비 0.15~0.20 → 누적 30만 토큰 쯤부터 봇이 치매 → "다음 장기챗할 때 이맘때쯤 슬슬 손요약해라" → NPC 리스트를 5만 토큰마다 수동으로 요약해서 로어북에 반영
SUMMARIZATION

이전 대화를 요약하여 압축하는 방식이다. HypaMemory V3는 메모리를 system 메시지로 주입하면서 매 턴 내용이 바뀌어 prefix 캐싱을 깨뜨린다. 요약마다 별도 LLM 호출도 필요하다.

prefix 캐싱 파괴

BACKGROUND

Stateful 에이전트가 어려운 이유

엔티티 — 같은 캐릭터를 뭐라 부르는지도 모른다

동일 캐릭터가 별명, 본명, 호칭으로 불리며 지칭이 달라진다. "루비아" = "Rubia"를 규칙으로는 잡을 수 없다. 업계 메모리 시스템(Mem0, Graphiti, LangMem)을 조사한 결과, 확실한 건 LLM 판단뿐이었다.

이벤트 — 뭘 저장하고 뭘 버릴지 기준이 없다

일시적 이벤트(싸움, 화가남)와 비가역적 이벤트(사망, 관계 변화)를 LLM이 구분할 수 없다. 잘못 구분하면 서사가 꼬인다.

입력 — 서비스마다 데이터가 전부 다르다

서비스마다 이미지, 수치, 로어북, 모든 형식의 입력이 달라진다. 하나의 파이프라인으로 모든 케이스를 처리할 수 없다.

언어 모델 — 요약이 한 번 틀리면 계속 틀린다

잘못된 요약 → 잘못된 맥락 주입 → 잘못된 응답이 다시 맥락에 쌓이는 악순환. 처음부터 잘못된 요약이 이후 대화를 계속 오염시킨다.

결국 다들 외부 스크립트나 로어북으로 구조화된 입력을 만들어서 Episodic Memory를 보완하고 있다. SAGA는 이걸 서버사이드에서 자동화한다.

INSIGHT

코딩 에이전트에서 찾은 패턴

Letta의 Stateful Agent는 강력하지만, 매 턴 3~5회 LLM 호출이라는 복잡도가 있다. 그런데 코딩 에이전트들(Claude Code, Cursor, Codex)은 수천 줄의 코드베이스를 넘나들면서, 메모리 구조는 의외로 단순하다. 전부 .md와 JSON 파일이다.

SystemContext ManagementPattern
Claude CodeCLAUDE.md, ~/.claude/memory/*.md마크다운 기반 지시 + 지속 메모리
Cursor.cursor/rules/*.mdc, .cursorrules프로젝트별 규칙 파일
CodexAGENTS.md, 마크다운 기반 지시에이전트 간 공유 컨텍스트
Letta (MemFS)memory/*.md + YAML frontmatter구조화된 메모리 블록

Letta의 MemFS가 특히 비슷하다. Memory Block(단일 문자열)에서 git 기반 마크다운 파일 시스템으로 진화한 구조인데, SAGA의 stable_prefix.md + live_state.md + YAML frontmatter와 거의 같다. 차이는 편집 주체다 — MemFS는 에이전트가 LLM 호출로 편집하고, SAGA는 코드 로직(Sub-B)이 밀리초 단위로 편집한다.

SAGA: Curator가 관리한 stable_prefix.md를 시스템 프롬프트에 주입 — DB 전체를 넣는 게 아니라, 정제된 .md 한 장만 넣는다

INSIGHT

구조화된 .md 요약

로어북, 에피소드 기록 같은 세부 데이터를 그대로 프롬프트에 넣으면, 시점이 다른 정보끼리 충돌하여 노이즈를 만들어낸다.

예시 1. 캐릭터 설정 모순

로어북: 엘리나는 착한 약초꾼이다

에피소드 — 턴 12: 엘리나가 점점 냉혹하게 변해들었다

예시 2. 상태 정보 충돌

에피소드 — 턴 8: 여관에서 쉬고 있다

에피소드 — 턴 11: 숲 한가운데서 길을 잃었다

이렇게 간단해 보이는 정보 차이가 누적되어 DB에 저장되면, LLM은 어떤 게 최신인지 판단할 수 없다. Curator가 정리한 "엘리나: 약초꾼이나 전투 가능", "현재 위치: 숲" — 충돌 없는 한 줄 요약이라 LLM이 바로 쓸 수 있고, 추론 부담도 줄어든다.

Curator는 정보를 더하는 게 아니라 정리한다 — 안 나오는 NPC는 빼고, 오래된 에피소드와 최신이 충돌하면 갱신하고, 여러 에피소드를 한 줄로 압축한다. 세부 데이터는 Curator 입력으로만 쓰고, LLM에는 정리된 .md만 넘긴다.

CORE DECISION

에이전트는 큐레이션에만

RP에서 에이전트의 판단력이 진짜 필요한 곳이 어디인지 생각해보면:

매 턴 서사 요약

"어떤 장면이었나, 누가 나왔나"

→ Flash 서사 요약으로 충분

매 턴 컨텍스트 조립

"어떤 로어북이 관련 있나"

→ 점수 기반 필터링으로 충분

N턴마다 서사 큐레이션

"이 모순은 의도된 건가, 이 복선은 회수해야 하나"

→ 이전 판단을 기억하고 이어가야 함. 에이전트가 필요

그래서 Letta를 메인 루프에서 빼고 Curator에만 배치했다. 코딩 에이전트가 Notepad로 작업 맥락을 자기관리하듯, Letta Curator는 Memory Block으로 큐레이션 판단 이력을 자기관리한다.

코딩 에이전트
Agent └ Notepad ← 자기관리 └ 작업 실행

"뭐가 중요하지?" "이전에 뭘 했지?"

SAGA 적용
Letta Curator └ Memory Block ← 자기편집 └ 서사 판단

"이 모순은 의도적인가?" "지난 큐레이션에서 뭘 결정했지?"

메인 루프는 LLM 0회(Sub-A) + 1회(내레이션)로 최소 지연이고, Curator의 다회 호출은 N턴마다 비동기라 유저에게 영향 없다.

PROMPT CACHING

슬라이딩 윈도우와 캐시

max token과 슬라이딩 윈도우

LLM API에는 한 번에 보낼 수 있는 토큰 한도(max context)가 있다. 대화가 길어져서 메시지 총량이 이 한도를 넘으면, 프론트엔드(RisuAI)가 오래된 메시지를 앞에서부터 잘라낸다. 이게 슬라이딩 윈도우다.

문제는 Anthropic의 프롬프트 캐싱이 prefix 해시 기반이라는 점이다. 메시지 배열의 앞부분이 이전 턴과 똑같으면 캐시 히트, 한 글자라도 다르면 전부 미스. 슬라이딩 윈도우가 앞쪽 메시지를 잘라내는 순간, 배열 자체가 바뀌니까 캐시가 통째로 날아간다.

즉, 대화가 길어질수록 캐시가 깨지기 쉽고, 캐시가 깨지면 매 턴 전체 토큰을 새로 처리해야 해서 비용과 지연이 같이 올라간다. SAGA는 이걸 세 가지로 잡는다.

System 메시지 변동

RisuAI가 Lorebook을 매 턴 시스템 프롬프트에 넣었다 뺐다 함. system 내용이 바뀌면 제일 앞(BP1) 캐시가 통째로 날아간다.

→ SystemStabilizer

슬라이딩 윈도우

토큰 한도 초과 → 앞쪽 메시지 삭제 → 메시지 배열이 달라짐 → prefix 전체 미스.

→ MessageCompressor

외부 윈도우 유실

RisuAI가 자체적으로 윈도우를 밀면 SAGA가 모르는 사이에 메시지가 사라진다. 맥락에 구멍이 뚫린다.

→ WindowRecovery

SYSTEM STABILIZER

시스템 프롬프트 고정

RisuAI는 활성화된 Lorebook 항목을 시스템 프롬프트에 동적으로 넣었다 뺐다 한다. 대화 중 조건이 바뀌면 시스템 메시지가 매 턴 달라지고, BP1 캐시가 깨진다.

Before (매 턴 변동)

// Turn 10: Lorebook "숲" 활성 system: 캐릭터 설정... + 이 숲에는 정령이 산다. // Turn 11: Lorebook "마을" 활성 system: 캐릭터 설정... + 마을 중앙에 분수가 있다. // → system 해시 변경 → BP1 미스
→

After (Stabilizer 적용)

// Turn 10~11: system 동일 system: 캐릭터 설정... ← 고정 // 바뀐 부분은 user prepend로 이동 user: [Lorebook: 마을 중앙에...] 실제 유저 입력... // → system 해시 불변 → BP1 히트

첫 턴의 시스템 프롬프트를 canonical로 저장해두고, 매 턴 현재 시스템과 paragraph 단위로 비교한다. 새로 추가된 부분(delta)만 떼어내서 마지막 user 메시지 앞에 붙인다. Jaccard 유사도가 임계값 아래로 떨어지면 캐릭터 자체가 바뀐 거니까 canonical을 새로 갱신한다.

MESSAGE COMPRESSOR

잘리기 전에 먼저 압축

대화가 길어지면 메시지 토큰이 한도에 가까워진다. RisuAI가 슬라이딩 윈도우로 앞쪽 메시지를 잘라내면 메시지 배열이 바뀌고 prefix 캐시가 전부 깨진다. MessageCompressor는 RisuAI가 자르기 전에 먼저 오래된 턴 쌍(user + assistant)을 요약 chunk로 교체한다. 한번 만든 chunk는 안 건드린다.

// 압축 전: 토큰 임계값 초과 { role: "user", content: "Turn 1 유저 입력..." } // 오래된 턴 { role: "assistant", content: "Turn 1 어시스턴트 응답..." } { role: "user", content: "Turn 2 유저 입력..." } { role: "assistant", content: "Turn 2 어시스턴트 응답..." } { role: "user", content: "Turn 3 유저 입력..." } { role: "assistant", content: "Turn 3 어시스턴트 응답..." } { role: "user", content: "Turn 4 유저 입력..." } // 유지 { role: "assistant", content: "Turn 4 어시스턴트 응답..." } // 압축 후: chunk로 교체 { role: "user", content: "[SAGA: 이전 대화 요약 1: Turn 1-3]" } { role: "assistant", content: "Turn 1: 소연이 검을 건넸다. ← turn_log에서 가져옴 Turn 2: 폐광 B2층으로 이동했다. Turn 3: 정령과 첫 조우.", cache_control: { type: "ephemeral" } ← BP2 고정 } { role: "user", content: "Turn 4 유저 입력..." } { role: "assistant", content: "Turn 4 어시스턴트 응답..." }

Chunk = 불변

Anthropic 캐시는 prefix 해시가 같아야 히트한다. chunk 내용이 한 글자라도 바뀌면 BP2부터 전부 미스. 그래서 chunk는 한번 만들면 안 건드린다.

새 턴이 압축되면 기존 chunk 뒤에 새 chunk를 붙일 뿐이다.

추가 LLM 호출 0회

chunk 요약 텍스트는 Sub-B가 매 턴 이미 만들어둔 turn_log.summary를 그대로 가져다 쓴다. 압축할 때 새로 요약을 돌리지 않는다.

turn_log가 없는 턴은 assistant 응답 앞 200자를 fallback으로 쓴다.

턴이 쌓이면서 일어나는 일

시점메시지 구조캐시 상태
Turn 1~20 system + 실제 메시지 20쌍 BP1(system) 캐시 히트
Turn 21 (임계값 초과) system + chunk_001(Turn 1~8) + 메시지 12쌍 BP1 히트 + BP2(chunk) 새로 캐시됨
Turn 30 (두 번째 압축) system + chunk_001 + chunk_002(Turn 9~16) + 메시지 14쌍 BP1 히트 + chunk_001 불변, 히트 + chunk_002 새로 캐시됨
Turn 100+ system + chunk_001~N + 메시지 ~10쌍 전체 chunk 불변 → prefix 캐시 계속 히트

WindowRecovery — Compressor가 미처 압축하기 전에 RisuAI가 윈도우를 밀어버리면, 첫 번째 메시지 해시 변화로 shift를 감지하고 잃어버린 턴의 turn_log를 모아 복구 블록(BP3)으로 주입한다. Compressor가 이미 처리한 범위는 건너뛴다.
