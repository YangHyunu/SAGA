AI 캐릭터 장기 대화에서 매 턴 해야 하는 일과 가끔 정리해야 하는 일을 분리한 FastAPI 리버스 프록시. 기억 검색·컨텍스트 조립·4필드 요약은 매 턴 빠르게 처리하고, 모순 검증·복선 회수·서사 구간 정리처럼 이전 판단을 이어받아야 하는 일만 Curator에 맡겼다. 저장소는 SQLite · ChromaDB · Markdown cache로 나눴고, 응답 이후에 처리해도 되는 작업은 비동기로 뺐다. 매 턴 바뀌는 기억은 캐시가 걸린 앞부분에 넣지 않도록 배치해 캐시 적중률 84.2%, 순비용 67.1% 절감을 확보했다. 같은 시드·시나리오 비교에서 LOCOMO judge 점수 2.02 → 3.12, LongMemEval 정답률 21.2% → 63.5%로 개선됐다.
1문제 정의
1.1 장기 기억의 문제

AI 캐릭터 챗에서는 대화가 길어질수록 캐릭터 설정과 관계 상태를 유지하기 어려워지는 문제가 반복된다. 직접 사용해보며 확인한 패턴은 크게 세 가지였다.

  • 성격 희석 — 일정 턴 이후 캐릭터 고유의 말투·반응 패턴이 약해지고, 일반 어시스턴트처럼 중립적인 응답이 늘어남
  • 관계 리셋 — "우리가 어제 싸운 일" 같은 관계 상태가 휘발, 유저가 매번 설명해야 함
  • 서사 유실 — 복선, 장소 이동, 등장 NPC 같은 구조적 맥락이 attention 범위를 벗어나면 사실상 소실

컨텍스트 윈도우에 전부 넣는 방식만으로는 해결되지 않는다. 200K 컨텍스트 모델에서도 중간 부분의 정보가 소실되는 lost-in-the-middle 현상이 보고되어 있어, 컨텍스트 길이를 늘리는 방식만으로는 장기 기억 문제를 안정적으로 해결하기 어렵다고 판단했다.

1.2 기존 솔루션의 한계

HypaMemory V3, Letta, mem0를 비교해보면 SAGA에서 피해야 할 지점이 비교적 분명했다.

솔루션 핵심 한계
HypaMemory V3 (RisuAI) 매 턴 관련 기억을 system 메시지 중간에 주입 → 프롬프트 앞부분이 매 턴 달라져 캐시가 깨짐. 실측 캐시 히트 12% 수준
Letta (MemGPT 계열) 자율 에이전트형 메모리로 턴당 3~5회 LLM 호출 (조회·업데이트·요약·결정). 레이턴시·비용 폭증
mem0 범용 Fact 저장에 최적화. NPC 관계·이벤트 체인 같은 서사 구조를 표현하기 어려움

그래서 SAGA에서는 기억을 잘 찾는 것과, 찾은 기억을 어디에 넣어야 캐시가 깨지지 않는지를 같이 봤다.

장기 대화 품질은 저장된 기억의 총량보다 현재 발화·관계 상태에 맞는 기억을 선별해 넣는 정확도에 더 크게 영향을 받았다. 검색이 맞아도 매 턴 컨텍스트를 다시 만드는 비용이 커지면 실제 세션에서는 쓰기 어렵다. 그래서 검색 결과의 품질뿐 아니라, 그 결과를 프롬프트 어디에 붙일지도 같은 문제로 다뤘다.
2아키텍처 설계
▼ 전제 · 프롬프트 캐시가 깨지는 위치를 피해야 했다 Anthropic API는 프롬프트의 앞부분이 이전 요청과 완전히 동일하면 그 부분을 재계산하지 않고 1/10 가격으로 재사용한다. 단, 앞부분의 바이트가 1개라도 바뀌면 캐시는 무효가 된다. SAGA에서 캐싱은 따로 붙인 기능이 아니라, 매 턴 바뀌는 내용을 어디에 둘지 정하게 만든 제약이었다. 그래서 매 턴 바뀌는 기억은 캐시 바깥에 두고, 시간이 지나며 쌓이는 모순과 복선은 Curator가 따로 정리한다.
2.1 설계 원칙

처음부터 모든 단계를 에이전트로 만들지는 않았다. 매 턴 반복되는 컨텍스트 조립·검색·요약·색인은 코드와 고정된 출력 형식으로 처리했다. 이전 판단을 이어받아야 하는 모순 검증·복선 회수·서사 구간 정리만 Memory Block을 가진 Curator에게 맡겼다. 고정 프롬프트 영역은 세션 내내 동일한 바이트를 유지하고, 매 턴 바뀌는 정보는 사용자 발화 직전 영역에만 추가했다. 저장소는 정확 조회(SQLite), 의미 검색(ChromaDB), 프롬프트 직렬화(Markdown cache)로 분리했다.

2.2 에이전트를 매 턴 돌리지 않은 이유

RP 장기 기억에서 매 턴 해야 할 일과 몇 턴에 한 번만 해도 되는 일이 섞이면 금방 비싸진다. 특히 Letta식 에이전트 루프를 매 턴 넣으면 조회·판단·수정·응답이 한꺼번에 돌면서 사용자가 전부 기다리게 된다.

작업 처리 방식
컨텍스트 조립 코드 로직 — Recent · Important · Similar 검색 결과를 RRF로 합치고 토큰 예산 안에 넣는다. LLM 판단이 필요하지 않다.
서사 요약 경량 LLM 1회 — 매 턴 4필드 JSON만 뽑는다. 이 결과를 압축·색인·DB 갱신·검증 시드가 같이 쓴다.
서사 큐레이션 Memory Block 에이전트 — "지난번에 이건 의도된 모순으로 봤다" 같은 결정을 기억해야 하므로 Curator가 필요하다.
결론: Letta 같은 에이전트 루프 전체를 메인 응답 경로에 넣지 않았다. 사용자가 기다리는 경로는 메인 LLM 1회로 끝내고, Curator는 N턴마다 뒤에서 실행해 narrative_summary, curation_decisions, contradiction_log를 자기편집한다.

2.5 Curator가 실제로 정리하는 것

내레이션·검색·요약·색인까지는 매 턴 처리로 충분했다. 문제는 50턴 이상 쌓인 뒤에 나왔다.

  • 캐릭터 동일성 — "이름 모를 남성"과 "MacNamara"가 동일 인물인지 식별
  • NPC 한·영 중복 — "Johnson"과 "존슨"이 별개 카드로 등록되는 문제
  • 타임라인 모순 — 같은 사건의 시각이 서로 다르게 굳어지는 경우
  • 서사 구간 정리 — 누적된 복선 중 회수된 것과 진행 중인 것을 구분

Sub-B의 4필드 요약은 매 턴의 사실만 추출하기 때문에 이런 누적 정리는 잡지 못한다. Curator는 N턴마다 전체 서사를 다시 훑고, 지난번에 내린 판단도 같이 본다. 그래서 같은 모순을 매번 새로 판단하지 않고, 지난 큐레이션의 결론을 이어받아 NPC 동일성·복선 상태·타임라인을 정리한다.

3구현하면서 부딪힌 문제들
3.1 검색 결과를 프롬프트 어디에 넣을까

매 턴 검색된 기억(관련 에피소드·관계 변화)을 프롬프트 어디에 넣을지가 첫 의사결정이었다. 첫 프로토타입에서는 검색 결과를 지시문 성격의 정보로 보고 system 메시지 끝에 추가했다.

문제는 캐시 로그를 찍어보면서 드러났다. 캐시 재사용 비율이 10% 초반에 머물렀다. 검색 결과는 매 턴 다른 내용이 들어오는데, 그걸 캐시가 걸린 앞부분 영역인 system 메시지에 넣으니 매 턴 앞부분이 새로워져서 캐시 전체가 무효화되고 있었다. 검색 결과는 유저 메시지 바로 앞으로 위치를 옮겼다. 유저 메시지는 어차피 매 턴 바뀌는 영역이라 캐시 바깥이다.

우려했던 품질 저하는 실험에서 유의미하지 않았다. 유저 발화 직전에 "참고 정보"로 붙는 편이 오히려 자연스럽게 반영되는 경향도 있었다. 이 한 번의 위치 변경으로 캐시 재사용 비율이 84%까지 올라갔다.

3.2 오래된 대화를 어떻게 밀어낼까

세션이 길어지면 컨텍스트가 폭증한다. 가장 쉬운 해법은 매 요청마다 과거 대화를 요약 모델로 돌려 압축본을 만드는 것이지만, 이러면 요약 결과가 매번 미세하게 달라진다. 같은 대화를 두 번 요약해도 토씨 하나는 바뀌기 마련이고, 그 바뀐 바이트가 캐시 영역에 들어가면 토큰을 줄이려는 압축이 오히려 캐시 효율을 떨어뜨리는 결과가 된다.

그래서 한 번 만든 요약은 이후 변경하지 않는다는 원칙으로 접근했다. 오래된 턴을 user·assistant 한 쌍 단위로 한 번만 요약하고, 이후에는 해당 요약 블록을 변경하지 않는 immutable summary chunk로 관리한다. 요약 내용도 새로 만들 필요가 없었다 — Sub-B가 매 턴 이미 만들어 둔 4필드 요약을 그대로 사용하기 때문에 LLM 호출이 추가로 발생하지 않는다.

대화가 길어지면 이런 봉인 블록이 계속 누적된다. 아래 그림이 시간 흐름에 따른 프롬프트 모양이다.

[턴 10] [system] [.md] [블록1] 📌캐시 경계 [최근 대화] [턴 30] [system] [.md] [블록1] 📌캐시 경계 [블록2] [블록3] [최근 대화] [턴 100] [system] [.md] [블록1] 📌캐시 경계 [블록2] ... [블록15] [최근 대화] └─ 이 영역 바이트는 턴이 쌓여도 완전히 동일 → 매 턴 캐시 재사용

블록은 뒤로만 추가되고, 한 번 만들어진 블록은 다시 변경되지 않는다. 그래서 캐시 경계를 가장 앞쪽의 블록1에 못 박아두면, 뒤에 블록이 몇 개가 붙든 앞부분의 바이트는 그대로라서 캐시가 유지된다. 이게 이 설계의 핵심이다.

  • 캐시 경계 위치 고정 — Anthropic 캐시 포인트(BP2)를 "가장 최근 블록"이 아닌 "맨 앞 블록(블록1)의 assistant 메시지"에 고정. 새 블록이 뒤에 붙어도 경계의 바이트 오프셋이 움직이지 않게 하기 위함.
  • 압축 임계치는 조립 후 측정 — "대화가 N 토큰 넘으면 압축" 판단을 원본 히스토리 기준으로 하면, 이미 블록으로 압축된 대화를 또 압축하는 오버 컴프레션이 발생한다. 해결은 현재 프롬프트를 실제로 조립한 뒤 그 결과의 토큰 수로 판단하는 것. 조립된 프롬프트는 이미 작아져 있으므로 임계치에 한참 미달 → 압축 트리거 안 걸림.
3.3 서사 요약을 한 번만 생성하기

초기 구조에서는 압축·검색 시드·색인·모순 검증 단계가 각각 요약을 따로 생성했고, 그 결과 턴당 경량 모델 호출이 4회 발생했다. 이 구조는 턴 수가 늘어날수록 보조 LLM 호출 비용이 선형으로 증가하는 문제가 있었고, 더 큰 문제는 같은 턴의 내용을 각 단계가 다르게 요약하면서 관계 업데이트와 에피소드 색인이 일치하지 않는 케이스가 나오기 시작했다는 점이었다.

Sub-B(Post-Turn)에서 4필드 구조로 한 번만 생성하고, 네 곳이 이 요약을 공유하는 구조로 바꿨다. 기준 요약이 하나로 통일되면서 일관성 문제가 사라졌고 LLM 호출도 1/4로 줄었다.

3.4 에이전트 루프를 메인 경로에서 뺀 이유

초기 프로토타입에서는 매 턴 해야 하는 일과 나중에 정리해도 되는 일이 섞여 있었다.

  • 매 턴 작업 — 컨텍스트 조립, 4필드 서사 요약. 규칙과 스키마로 처리 가능. 이전 판단을 기억할 필요 없음.
  • N턴마다 작업 — 모순 검증, 서사 구간 재정리, 로어 추출. "지난 번엔 이 모순을 의도된 거라고 결정했다"는 이력을 참조해야 일관성이 잡힘.

처음에는 두 작업을 같은 LLM 호출 흐름에 끼워 넣었다. 그러자 매 턴 호출 수가 늘었고, 큐레이션 판단도 매번 처음부터 다시 시작됐다. 같은 모순을 다른 결론으로 처리하는 경우도 있었다.

그래서 둘을 분리했다. 매 턴 작업은 메인 루프에 남기되 LLM 1회로 제한하고, 큐레이션은 별도 트랙으로 빼서 N턴마다 비동기로 실행했다. 지난 결정을 이어받기 위해 큐레이션 자리에는 Memory Block을 가진 에이전트(Letta)를 두고, 매 호출마다 narrative_summary·curation_decisions·contradiction_log를 자기편집하도록 했다.

메인 루프는 LLM 1회로 줄어 사용자가 기다리는 시간이 안정됐고, 큐레이션은 이전 판단을 보면서 결론을 이어갈 수 있었다. Letta를 메인 루프가 아니라 큐레이션 자리에만 쓴 이유가 여기 있다.

4.2 벤치마크 방법
  • LOCOMO (ACL 2024) — 장기 대화 QA 벤치마크. 단일 세션 내 50~100턴에서 과거 사실 회상 능력을 측정. Judge 점수(1~5)로 평가, SAGA는 judge 2.02 → 3.12로 향상 (+54%)
  • LongMemEval (ICLR 2025) — 시간 간격을 둔 여러 세션에서 사실·관계·시간 추론 능력 측정. SAGA는 21.2% → 63.5% (+42.3%p)
  • multi-hop 세부 지표 — "A가 B에게 준 물건을 C가 알고 있는가" 같은 2-hop 이상 질문에서 4.71/5 달성
벤치마크의 한계 캐릭터 챗의 품질을 LOCOMO와 LongMemEval만으로 온전히 평가하기는 어렵다. 두 벤치마크는 "대화에 포함된 사실을 얼마나 정확히 회상하는가"를 정답이 정해진 QA로 측정하지만, 실제 롤플레이는 유저의 한 마디로 장소가 바뀌고 새 NPC가 튀어나오며 설정이 즉흥적으로 갱신되는 non-stationary 환경이다. 말투 일관성·관계 밀도·상황 판단 같은 정성 요소도 크게 작용한다. 이 한계를 인정하고, SAGA는 정량 벤치마크를 "큐레이션 정확도의 하한 체크"로 사용하며, 품질 판단은 동일 캐릭터로 수백 턴을 누적한 장기 재현성 테스트로 보완한다.
4.3 캐시 · 비용 분석

Anthropic API의 캐시 재사용 토큰과 캐시 쓰기 토큰을 분리 집계하여 실제 청구액 기준으로 비교했다. SAGA OFF(기본 RisuAI) 대비 ON 상태에서:

  • 캐시 재사용 비율 84.2% — 입력 토큰의 84%가 캐시 히트로 처리되어 1/10 단가 적용
  • steady-state 86.8% — 세션 초반 5턴을 제외한 안정 구간에서의 수치. 실서비스 조건에 가까움
  • 순비용 67.1% 절감 — 캐시 할인 + 프롬프트 단축 효과 합산
4.4 정성 평가 (장기 사용 검증)

동일 캐릭터·시나리오로 수백 턴 누적 세션을 반복해 재현성을 확인했다. max context 180K, HypaMemory OFF + SAGA ON 설정 기준으로 관찰한 변화는 다음과 같다.

관찰 내용 (HypaMemory ON 비교 기준):
· 장기 누적 세션(30만+ 토큰)에서도 동일 캐릭터·시나리오에서 말투 유지와 과거 사건 참조 빈도가 더 안정적으로 관찰됨
· "그때 ○○이 했던 말" 류 과거 참조 질문에 구체적 에피소드 회상이 가능. 기존 설정에서는 과거 사건을 누락하거나 실제 대화에 없던 내용을 생성하는 경우가 자주 발생했음
· 응답 레이턴시 체감 변화 없음 — 기억 갱신이 백그라운드로 분리되어 유저 응답 경로에 포함되지 않음
5포지션 변경과 다음 단계
5.1 v1의 도달점

SAGA는 중앙 처리부(매 턴 컨텍스트 조립 · 비동기 Sub-B · Memory Block Curator · 메모리 3계층 · 캐시 보존)와 벤치마크 검증(LOCOMO · LongMemEval) 단계까지 완성됐다. 본래 목표였던 커뮤니티 플러그인 모듈로의 패키징 — Docker 이미지화, 설치 경로 간소화, RisuAI·SillyTavern 통합 가이드 작성 — 단계는 진입하지 않은 상태에서 v1을 마무리했다.

그 외 내부 정리·리팩터링 작업은 다음 항목까지 진행됐다.

  • chat_handler.py 책임 분리 (완료) — 512줄 단일 파일을 세션 관리·캐시 마킹·파이프라인 오케스트레이션·후처리 4파일로 분해
  • WindowRecovery 제거 (완료) — MessageCompressor가 완전 대체하면서 파이프라인 단계 1개 축소
  • ChromaDB 용도 재정렬 (완료) — Lorebook을 SQLite 단일 소스로 이관, ChromaDB는 에피소드 전용으로 역할 명확화
  • Curator 책임 분리 (보류) — 모순 검증과 lore 생성 분해 작업이 v1 마무리 시점에 보류
5.2 포지션 변경

v1에서는 "매 턴 작업은 가볍게 처리하고, 판단 이력이 필요한 큐레이션만 에이전트에 맡긴다"는 방향을 내부 평가로 확인하는 데 집중했다 (캐시 적중률 84.2%, 비용 67.1% 절감). 패키징과 배포 자동화는 v1 범위에서 후속 과제로 남겼다.

이후 검증한 프롬프트 캐싱·장기 맥락 관리 구조는 다른 LLM 시스템 설계에 재사용하고 있다.
